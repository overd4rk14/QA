# Тест-план для раздела «Корпоративное обучение» сайта «Школа 21»

## 1) Вступление

Настоящий документ является тест-планом для раздела «Корпоративное обучение» сайта «Школа 21» (веб-версия). Его цель — определить объём, стратегию и ресурсы для тестирования данного раздела, который представляет услуги компании «Школа 21» для бизнес-клиентов. План служит основой для согласования работ между участниками проекта и используется для контроля и отчётности в процессе тестирования.

## 2) Функциональности, которые будут протестированы

*   **Основной контент страницы:**
    *   Заголовки и текстовые блоки, описывающие форматы корпоративного обучения (групповые интенсивы, цифровая грамотность для команд).
    *   Ключевые ценностные предложения для бизнеса.
*   **Форма обратной связи / заявки:**
    *   Поля для ввода данных компании и контактной информации.
    *   Кнопка отправки формы.
    *   Валидация введённых данных (проверка обязательных полей, формата email, номера телефона).
*   **Навигация и ссылки:**
    *   Работоспособность всех внутренних ссылок внутри раздела.
    *   Кнопки призыва к действию (CTA), например, «Оставить заявку».
    *   Корректность переходов на связанные разделы сайта (например, «Компаниям»).
*   **Презентационные элементы:**
    *   Корректность отображения изображений, иконок, графиков или видео-презентаций.
    *   Структура и читаемость текста.

## 3) НЕ будут протестированы

*   **Интеграция с CRM-системой:** Логика обработки и сохранения заявки на стороне сервера после отправки формы.
*   **Административная панель (CMS):** Функциональность добавления или редактирования контента раздела.
*   **Личный кабинет корпоративного клиента:** Процессы, требующие авторизации и связанные с управлением заказами или обучением сотрудников.
*   **Смежные, но отдельные разделы сайта:** Такие как «Кто мы», «Вузам» или «Как мы учим» — они являются отдельными объектами тестирования.
*   **Производительность и нагрузка (load/stress):** Проверка поведения сайта под высокой нагрузкой, так как для этого требуются специальные инструменты и выделенное стенд-окружение.
*   **API внешних сервисов:** Работа сторонних сервисов, если они используются (например, сервисы аналитики или чат-боты).

## 4) Критерии приёмки (Pass/Fail)

**PASS, если:**
1.  Все основные функциональные блоки (контент, форма, навигация) работают корректно.
2.  Форма обратной связи:
    *   Отправляется при заполнении валидными данными.
    *   Корректно отображает сообщения об ошибках при попытке отправки с пустыми обязательными полями или неверным форматом данных (email, телефон).
3.  Все ссылки на странице ведут на ожидаемые разделы (внутренние) или корректные внешние ресурсы, нет битых ссылок (ошибок 404).
4.  Страница корректно отображается и все элементы интерфейса кликабельны в следующих окружениях:
    *   OperaGX (последняя стабильная версия) на Windows.
5.  Контент страницы не содержит критических орфографических или грамматических ошибок, а информация выглядит актуальной.
6.  Нет открытых дефектов с уровнем серьезности **Blocker** или **Critical**.

**FAIL, если:**
*   Не выполняется хотя бы один из пунктов критериев **PASS**.
*   Обнаружена ошибка, полностью блокирующая ключевой пользовательский сценарий (например, невозможность отправить заявку).

## 5) Задачи тестировщиков

1.  **Анализ требований:** Ознакомление с разделом, понимание целевой аудитории и ключевых пользовательских сценариев.
2.  **Создание тестовой документации:** Разработка чек-листов и/или тест-кейсов для проверки функционала из раздела 2 данного плана.
3.  **Выполнение тестирования:**
    *   Проведение функционального тестирования.
    *   Проведение кросс-браузерного тестирования согласно разделу 9.
    *   Проверка адаптивности верстки (на разных разрешениях экрана).
    *   Проведение исследовательского (exploratory) тестирования для выявления неочевидных проблем.
4.  **Документирование дефектов:** Четкое и подробное описание найденных ошибок в баг-трекинговой системе (например, Jira).
5.  **Ретестинг:** Проверка исправленных разработчиками дефектов.
6.  **Подготовка отчётности:** Формирование отчёта о результатах тестирования.

## 6) Потребности в окружении

*   **Операционные системы:** Windows 10/11.
*   **Браузеры (стабильные версии):**
    *   OperaGX
*   **Разрешения экрана для проверки адаптивности:** 1920x1080 (референсное), 1366x768.
*   **Тестовый стенд:** Доступ к странице раздела «Корпоративное обучение» на тестовом/промежуточном (staging) окружении.
*   **Тестовые данные:** Набор валидных и невалидных данных для заполнения полей формы (названия компаний, email, телефоны).
*   **Инструменты:** DevTools браузера, средство для создания скриншотов.

## 7) Персонал и обучение

*   **Состав команды:** 1 тестировщик (ручное тестирование).
*   **Ответственности:** Тестировщик отвечает за выполнение всех задач, перечисленных в разделе 8.
*   **Обучение/брифинг:** Перед началом работ тестировщик должен быть ознакомлен с:
    *   Данным тест-планом.
    *   Общими принципами работы и позиционированием «Школы 21».
    *   Процедурой документирования дефектов, принятой в проекте.

## 8) Артефакты тестирования

По итогам тестирования будут созданы и предоставлены следующие артефакты:
1.  **Чек-листы / тест-кейсы**, использованные в процессе проверки.
2.  **Отчеты о выполнении прогонов** с указанием пройденных/непройденных проверок.
3.  **Отчеты о дефектах** в баг-трекинговой системе.
4.  **Итоговый отчет о тестировании**, содержащий сводную информацию о проведенной работе, метриках (например, процент успешных тестов) и рекомендации по выпуску.
