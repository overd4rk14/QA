
> **Выбранное веб-приложение:** **Авито** (avito.ru) — крупнейшая российская доска объявлений. Пользователи размещают и ищут объявления о продаже товаров, недвижимости, услугах и вакансиях.

![блок-схема](misc/images/avito.png)

### Основные компоненты архитектуры

| Компонент | Для чего нужен |
|-----------|---------------|
| Балансировщик нагрузки | Распределяет входящий трафик между серверами, чтобы сайт не перегружался |
| Сервис объявлений | Отвечает за создание, редактирование и удаление объявлений |
| Сервис поиска | Позволяет быстро искать объявления по разным параметрам |
| База данных (PostgreSQL) | Хранит все данные: объявления, профили, историю |
| Redis | Хранит временные данные и ускоряет работу сайта |
| S3-хранилище | Хранит фотографии из объявлений |

## Примеры сбоев и тестирования

| Компонент | Возможные сбои | Как это видит пользователь | Какой тест нужен |
|-----------|----------------|---------------------------|-----------------|
| Сервис поиска | 1. Перегрузка — поиск работает медленно или с ошибкой <br> 2. Несовпадение с базой — показываются устаревшие объявления | 1. Долго грузится поиск или ошибка <br> 2. В поиске есть удалённые объявления | 1. Нагрузочное тестирование <br> 2. Проверка согласованности данных |
| База данных | 1. Основная база недоступна — нельзя добавить объявление <br> 2. Медленные запросы — сайт тормозит | 1. Ошибка при добавлении объявления <br> 2. Страницы долго открываются | 1. Тест отказоустойчивости <br> 2. Тест производительности |

## Горячий и холодный резерв

- **Горячий резерв** — запасной сервер всегда работает и готов сразу заменить основной, если тот сломается. Данные почти не теряются, переключение быстрое, но это дороже.
- **Холодный резерв** — запасной сервер выключен. Если основной сломался, нужно вручную включить резерв, восстановить данные из копии. Это дешевле, но дольше и возможна потеря данных.

---

## Сравнение резервов

| Критерий | Горячий резерв | Холодный резерв |
|----------|---------------|----------------|
| Готовность | Всегда готов | Нужно запускать вручную |
| Синхронизация | Почти сразу | С задержкой, возможна потеря данных |
| Время переключения | Секунды | Минуты или часы |
| Стоимость | Дорого | Дешево |
| Плюсы | Быстро, нет потери данных | Просто и дешево |
| Минусы | Дорого, сложно | Долго, возможна потеря данных |

## Пример теста отказоустойчивости (web-тестирование, горячий резерв)

**Что проверяем:** Если основной сервер базы данных сломается, сайт быстро переключится на запасной, и пользователи смогут продолжать пользоваться сайтом без серьёзных проблем.

**Как тестируем (web-ориентированный сценарий):**
1. Открываем сайт Авито в браузере (несколько пользователей одновременно).
2. Проверяем, что сайт работает: можно искать объявления, переходить по страницам, авторизоваться.
3. Имитируем сбой основного сервера базы данных (например, отключаем его на тестовом стенде).
4. Следим, чтобы сайт не "упал": страницы должны открываться, поиск и просмотр объявлений должны работать, пусть даже чуть медленнее.
5. Пробуем добавить новое объявление — оно должно появиться в поиске.
6. После восстановления основного сервера убеждаемся, что все данные на месте, ничего не пропало.

**Что отслеживаем:**
- Время, за которое сайт восстановил работу (не больше 30 секунд)
- Нет потери объявлений
- Ошибок на сайте во время сбоя — не больше 1%
- Скорость работы сайта возвращается к норме за минуту

## Метрики для отслеживания в тесте (Web terminology)

| Категория | Метрика  | Описание | Как измерять | Целевое значение |
|-----------|------------------|----------|--------------|------------------|
| **Доступность** | Availability (SLA), RTO (Recovery Time Objective) | Время, за которое сайт становится снова доступен после сбоя. Показывает, насколько быстро система восстанавливает сервис для пользователей. | Логи, таймер, мониторинг (например, uptime-боты, Prometheus) | RTO ≤ 30 сек, Availability ≥ 99% |
| **Целостность данных** | Data Consistency, RPO (Recovery Point Objective) | Потеря или дублирование пользовательских данных (например, объявлений) после сбоя. | Сравнение данных до и после сбоя, UI-проверка, сравнение логов | RPO = 0 (нет потери данных) |
| **Производительность** | Response Time (Время отклика), p95 Latency | Время загрузки ключевых страниц (поиска, карточки объявления, публикации). p95 — 95-й перцентиль, отражает опыт большинства пользователей. | WebPageTest, Lighthouse, DevTools, APM, k6, Prometheus | p95 ≤ 1 сек |
| **Пропускная способность** | Throughput (RPS — Requests per Second) | Количество успешно обработанных запросов в секунду во время и после сбоя. | Мониторинг, Grafana, Prometheus | Не ниже 80% от baseline |
| **Ошибки** | Error Rate (HTTP 5xx, 4xx) | Доля неуспешных запросов (ошибки сервера/клиента) во время сбоя и восстановления. | Логи, мониторинг, консоль браузера, Sentry | ≤ 1% |
| **Пользовательский опыт** | User Experience (UX) | Сайт визуально доступен, основные сценарии (поиск, просмотр, публикация) работают, нет критических багов в интерфейсе. | Ручное тестирование, чек-листы, автоматизация UI | Все критичные сценарии проходят |

### Описание ключевых метрик:
- **Availability (Доступность)** — процент времени, когда сайт доступен для пользователей. Важно для SLA.
- **RTO (Recovery Time Objective)** — максимальное допустимое время восстановления после сбоя.
- **RPO (Recovery Point Objective)** — максимальное допустимое количество потерянных данных (например, объявлений) после сбоя.
- **Response Time / Latency** — время, за которое пользователь получает ответ от сайта (страница, поиск, публикация).
- **Throughput (Пропускная способность)** — сколько запросов в секунду сайт может обработать без деградации.
- **Error Rate** — процент неуспешных запросов (ошибки 5xx, 4xx) от общего числа.
- **User Experience** — субъективная оценка удобства и работоспособности сайта глазами пользователя.

---

**Почему выбраны такие метрики:**  
Они позволяют объективно оценить, насколько сайт надёжен, быстро восстанавливается после сбоя, не теряет данные и остаётся удобным для пользователей. Использование профессиональных терминов облегчает коммуникацию с разработчиками и заказчиками.
